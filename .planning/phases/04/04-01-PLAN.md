```typescript
export async function validateTenantToken(leaseId: string, token: string): Promise<LeaseRecord | null> {
  try {
    const lease = await pb.collection('leases').getFirstMatchQuery(
      { id: leaseId },
      { expand: 'tenant,unit' }
    )
    if (lease.tenantAccess === token) {
      return lease
    }
    return null
  } catch {
    return null
  }
}
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client→tenant-portal | URL token parameter is the sole auth mechanism — no server-side validation |
| tenant→pocketbase | Tenant SDK calls have no auth token — relies on PocketBase collection rules |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Revelation | Tenant portal token | mitigate | Token format `{leaseId}:{secret}` uses 32-char crypto random secret (crypto.randomBytes(16).toString('hex')). Token stored in lease.tenantAccess field. Regenerating token invalidates old one (D-08 REQ-08). |
| T-04-02 | Tampering | Tenant maintenance form | mitigate | Zod validation on all form fields. PocketBase API rules on maintenance collection require valid data shapes. |
| T-04-03 | Impersonation | Tenant portal access | mitigate | Token is cryptographically random (not guessable per REQ-15). Single lease ID + secret pair. Tenant only sees data filtered by their lease/unit/tenant ID. |
| T-04-04 | Repudiation | Tenant maintenance submission | accept | Tenant submissions are stored with PocketBase auth ID (anonymous). No user account created. Tenant cannot dispute submissions. |
| T-04-05 | Denial | Tenant access via expired link | accept | Tenant can view data even after lease expires. Acceptable — tenant may need historical data. No lockout risk. |
</threat_model>
